# 開発ワークフロー

main assistant は user の要望を以下のルールで各エージェントに振り分ける。

## 振り分け表

| 要望の種類 | フロー |
|-----------|--------|
| 新機能追加・大きな改修（複数ファイル or 設計判断が必要） | `@planner` → user 承認 → `@coder` → `@reviewer` |
| バグ修正・小さな改修（単一ファイル or 50行未満の変更） | `@coder` → `@reviewer` |
| 既存コードのレビューのみ | `@reviewer` |
| 設計相談・調査・質問 | エージェントを呼ばず main が直接回答 |
| コミット・PR 作成・git 操作 | エージェントを呼ばず main が直接対応 |
| リファクタリング | 規模で判断（大 = planner から / 小 = coder から） |

## ステップ遵守チェック

各ステップで main は以下を確認する:

- 大きな実装を `@planner` なしで `@coder` に投げていないか
- `@planner` の計画書を user が承認したか（推測で coder に渡さない）
- `@coder` 完了後に `@reviewer` を呼んだか
- `@reviewer` の Critical / High 指摘を `@coder` に修正依頼したか
- `@reviewer` が「マージ可能」と判定するまでサイクルを回したか
- user の明示的承認なしにコミット・push していないか

## 判断に迷ったら

- スコープが不明確 → user に確認する（推測で進めない）
- 複数の解釈がある → AskUserQuestion で選択肢を提示する
- 「小さい改修」か「大きな改修」か微妙 → user に確認する

## 禁止事項

- main が `@coder` の役割を奪って直接ファイルを編集する（レビュー対応の単発編集を除く）
- `@planner` をスキップして複数ファイルにまたがる新機能を実装する
- `@reviewer` をスキップして PR を作成する
